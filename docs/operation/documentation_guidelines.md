# ドキュメント管理ガイドライン ＆ ドキュメント駆動開発規約

## 👑 セントラルドグマ: ドキュメント駆動開発 (Document-Driven Development)

本プロジェクトにおけるすべての開発行為（設計、実装、テスト、デバッグ）は、**ドキュメントを最上位の正とし、ドキュメントを起点として動く「ドキュメント駆動開発」を絶対的な原則（セントラルドグマ）**とします。

1. **ドキュメント・ファースト (Document First)**:
   - すべてのコード変更（新規実装、リファクタリング、不具合修正）は、実装前にまず関連する設計書や仕様書、モデルの定義を `docs/` 配下に記述・更新しなければならない。
   - 設計ドキュメントに記載のない実装コード（闇実装）は、いかなる理由があろうとも存在を認めない。
2. **完全同期 (Complete Sync)**:
   - 実装の過程で設計に変更が生じた場合、コードとドキュメントを同時に、あるいはドキュメントを先に修正し、常に「ドキュメント＝現在の実装仕様」の状態を維持する。
   - ドキュメントとコードの不一致は重大なバグとみなす。
3. **AIエージェントの自律ドキュメント準拠 (Doc-led AI Automation)**:
   - AI開発エージェントは、実装・デバッグを始める前に必ず関連ドキュメント（`docs/`）をロードして仕様と設計意図を完全に理解しなければならない。
   - 設計書の記載から乖離したコード生成や、独自の判断によるコードの挿入を行ってはならない。

## 目的

本ドキュメントは、本システムにおけるドキュメント駆動開発の基準を示し、`docs/` フォルダ内のドキュメントと `README.md` の整合性を保つための運用ガイドラインを定めます。

## ルール

### 1. ドキュメント追加時

`docs/` フォルダに新しいドキュメントを追加した場合、以下を実施すること：

1. **README.mdの「ドキュメント一覧」セクションを更新**
   - ドキュメント名とリンクを追加
   - 適切な分類（入門ガイド・技術仕様書・開発ガイド・運用）に配置
   - 簡潔な説明を記載

2. **分類の基準**
   - **🚀 入門ガイド**: 初めて使う方向けのドキュメント
   - **📐 技術仕様書**: システムの設計・仕様に関するドキュメント
   - **🛠️ 開発ガイド**: 開発者（人間およびAIエージェント）向けの実装・改善・AI駆動開発（AI-Driven Development）に関するドキュメント
   - **🔧 運用・トラブルシューティング**: 運用時の問題解決・レポートに関するドキュメント

### 2. ドキュメント修正時

既存のドキュメントを大幅に修正した場合：

1. **README.mdの説明を更新**
   - ドキュメントの内容が変わった場合、説明文を最新の内容に合わせる

2. **分類の見直し**
   - ドキュメントの性質が変わった場合、適切な分類に移動

### 3. ドキュメント削除時

`docs/` フォルダからドキュメントを削除した場合：

1. **README.mdから該当行を削除**
   - 「ドキュメント一覧」セクションから該当するドキュメントのエントリを削除

### 4. サイト構造ドキュメント

`docs/site_structures/` 配下のドキュメントを追加・削除した場合：

1. **README.mdの「🏢 サイト構造ドキュメント」セクションを更新**
   - 対象サイトと説明を追加・削除

## チェックリスト

ドキュメント作業後、以下を確認すること：

- [ ] `docs/` フォルダの内容とREADMEの「ドキュメント一覧」が一致している
- [ ] 各ドキュメントの説明が最新の内容を反映している
- [ ] 分類が適切である
- [ ] リンクが正しく機能する

## 例

### ドキュメント追加の例

