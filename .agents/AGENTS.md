# プロジェクト共通開発ルール

## サブエージェント・メインエージェントのモデル利用方針
- 簡単な作業（ブラウザでの単純な巡回、情報の収集、UI動作確認など）は、コスト効率の高い（安い）AIモデル（Gemini 3.5 Flash 等）のサブエージェントを利用して処理させる。
- 高コストなモデル（Claude Opus 等）は使用せず、常にコスト効率の高いモデル（Gemini 3.5 Flash等）を利用することを遵守する。
- 開発者（AIエージェント）自身も作業を行う際は常にこの方針を念頭に置き、ユーザーからのすべての指示でコスト効率の高いモデルの使用を徹底する。

## トークン消費の抑制（Micro-Diff方針）
- コード修正を提案する際は、ファイル全体を再出力するような高コストな書き換えは避け、最小限の変更ブロック（Micro-Diffまたは特定箇所のReplace）に絞ってコード生成を行う。
- 未使用のライブラリや不要なデッドコードは、コンテキスト肥大化を防ぐために積極的に自動削除（`ruff check --fix` 等によるクリーンアップ）を行う。

## Slack通知の疎通自己チェックルール
- Slackの通知先チャンネル名（スペルなど）の変更、あるいはBotがチャンネルに招待されていないことが原因で通知が不達（`channel_not_found` 等）になる問題を防ぐため、クローリングおよび価格推定パイプラインの起動前（Step 0）に必ずSlackの疎通自己チェック（`check_slack_connection.py`）を実行して検証すること。

## コンテナ変更時の事実検証ルール
- Dockerfile、docker-compose、devcontainerなどのコンテナ構成定義や依存関係を変更した際は、単にビルド成功ログを確認するだけではなく、コンテナが実際に新規ビルド・再作成された事実を裏付けすること。
- また、変更内容（追加したコマンドの存在やマウントされたディレクトリなど）が、実際に起動しているコンテナ内に正しく反映されているかを、事前にコマンド実行（例: `docker-compose exec -T app command -v ps` 等）で自己検証すること。
- 上記による事実の裏付けが取れるまでは、推測のみでユーザーに環境の切り替えや動作確認を求めてはならない。

## パス指定の原則（ホスト・コンテナ間依存の排除）
- ログや一時ファイルなどの保存先パスを記述する際は、`/app/` などの特定の絶対パスをハードコードしてはならない。
- 常に `__file__` などから動的にプロジェクトのルートディレクトリを算出し、`os.path.join` で相対的に組み立てること。これによりホスト側（Windows等のホストOS）とコンテナ側（Docker）の両方の実行環境で正しくパスが動作することを保証すること。
